# 错误输入如何被验证、修正与局部修改

更新日期：2026-04-15

这份说明文档回答一个关键问题：如果使用者本人的理解、选股逻辑或交易前提本身就是错的，框架能不能帮助修正？

短答案是：可以帮助暴露、修正、降权和限定适用范围，但它不是一个自动宣布“你错了”的裁判系统。

> 这个框架的核心哲学是：系统是用户的记忆增强外脑，而不是一个会惩罚错误输入的 oracle。

## 一、先说边界

当前框架不是：

- 自动判断输入真假的知识裁判
- 自动替代用户决策的交易代理
- 一次性删掉“错误思路”的强制过滤器

当前框架能做的是：

- 用历史失败案例与结构化证据，把矛盾暴露出来
- 把错误后的复盘固化成“纠错记忆”
- 允许用户局部修改标签、质量分、市场阶段和禁止条件
- 通过 bandit 与技能固化门槛，让错误路径逐步降权

## 二、机制总览

| 机制 | 类型 | 作用 |
| --- | --- | --- |
| 历史失败案例召回 | 隐式验证 | 相似场景下优先召回亏损记录，让用户自己看到反证 |
| 复盘纠错记忆 | 显式修正 | 把错误触发条件、损失和教训写入下一次检索 |
| 手动编辑记忆 / 技能卡 | 局部修改 | 直接改标签、质量分、市场阶段、禁止条件 |
| Bandit 效用衰减 | 自动演化 | 错误路径效用下降后，提醒排名自然变低 |
| 技能固化门槛 | 保守验证 | 不稳定或负收益路径难以固化成 skill |
| 社区交叉验证（未来） | 群体验证 | 多个用户共享记忆后，可识别高风险共性模式 |

## 三、隐式验证：让历史矛盾自己浮出来

当用户输入一个思路，例如：

> 在下跌趋势里追突破

系统不会直接粗暴地回答“这一定错”。它会先根据当前查询去召回：

- 相似标签的历史交易
- 同类 market stage 下的失败案例
- 已经沉淀为 skill card 的相关路径
- 与该路径相关的错误簇、风险簇、情绪簇

如果历史记忆里大量失败案例都带着“追突破 + 下跌趋势 + 缩量”的结构，那么这些记忆会在前排出现。此时系统给用户的是一种基于历史证据的验证信号，而不是武断裁定。

这就是“隐式验证”：

- 不直接给出神谕
- 但把反证放到眼前
- 让用户基于自己的历史账本修正理解

## 四、显式修正：把错误复盘写成新的纠错记忆

如果用户真的执行了错误思路，系统支持把这次失败复盘沉淀为纠错记忆。典型内容包括：

- 错误触发条件：例如“追突破 + 下跌趋势 + 缩量”
- 造成的损失：例如“买后两日继续下杀，止损离场”
- 教训总结：例如“下跌阶段不应把局部反抽误判成新主升”

一旦写入，这条复盘会进入：

- `memory_cells`
- 对应的 `scene`
- 相关的 `hyperedge`
- 后续 reminder 与 query 的候选集合

此后系统再次碰到相似查询时，这条“纠错记忆”就会被召回，从而形成负向反馈闭环。

## 五、局部修改机制：不是整库推倒，而是精修

本框架是本地优先的，所以用户对记忆层拥有完全控制权。你不需要等系统“学会”，就可以直接局部修正。

### 1. 修改记忆单元

你可以直接修订某条记忆的：

- 标签
- 市场阶段
- 质量分
- correction note

示例：

```powershell
python .\finance-journal-orchestrator\scripts\finance_journal_cli.py memory revise memory_trade_xxx `
  --add-tags error_cluster:trend_conflict `
  --quality-score -8 `
  --market-stage downtrend `
  --correction-note "下跌趋势追突破，这个思路在这里是错的"
```

这样做的意义是：

- 纠正原先错误的归因
- 让这条记忆在后续检索中的排序下降
- 把“错误原因”作为显式证据保存下来

### 2. 删除或降权低质量记忆

不是每一条历史记录都值得反复召回。

如果某条记忆本质上是当时误判、噪声记录、错误归因，用户可以通过降低 `quality_score` 的方式让它在检索中更靠后，而不是粗暴删除整段历史。

这保留了可追溯性，也避免反复被错误经验误导。

### 3. 修改技能卡的禁止条件

有些思路不是“永远错误”，而是“在某些场景下不该用”。这时更适合在 skill card 上补充 `do_not_use_when`。

示例：

```powershell
python .\finance-journal-orchestrator\scripts\finance_journal_cli.py memory skill-edit skill_quality_path_xxx `
  --add-do-not-use-when "当市场缩量下跌时禁止使用"
```

这会带来两个好处：

- 不是把整张技能卡删除，而是收紧适用边界
- 下次召回技能卡时，系统可以同时把“可用条件”和“禁用条件”一起展示给用户

## 六、Bandit 层如何自动降低错误思路优先级

当前框架仍保留 bandit 风格的上层演化逻辑。记忆层负责“召回候选”，bandit 层负责“优先级排序”。

对候选提醒或行动 $a$，可以写成：

$$
U(a)=\hat{\mu}_a+\lambda\sqrt{\frac{\ln(1+N)}{1+n_a}}
$$

其中：

- $\hat{\mu}_a$ 是历史效用估计
- $n_a$ 是该路径被使用的次数
- $N$ 是总交互次数

如果某条基于错误选股思路的提醒：

- 反复被用户忽略
- 被采纳后又在复盘里标记为亏损
- 持续伴随错误簇标签出现

那么它的效用估计就会下降，未来排序自然下滑。这个过程是“渐进式降权”，不是“立即消失”。

## 七、技能固化前的保守验证

系统不会轻易把一条错误路径固化成可分享 skill。

论文式目标可以写成：

$$
\mathrm{skillize}(p)=\mathbb{1}[n_p\ge n_{\min}]\cdot\mathbb{1}[\bar{r}_p-\eta\sigma_p>\tau]
$$

直觉上就是：

- 样本不够，不固化
- 平均效用不够，不固化
- 波动太大、过于不稳定，也不固化

因此，一个错误思路即便出现过几次，也很难被提升为“社区可复用技能”。

## 八、社区层的未来交叉验证

未来如果进入社区互联层，记忆卡与 skill card 会进一步具备交叉验证能力。

可能的机制包括：

- 多个用户都把同类思路标记为高风险时，系统提示“社区层高风险模式”。
- 社区成员可对 skill card 打分、评论、补充失败边界。
- 某个技能卡如果只在少数场景有效，也可以被社区共同收紧 `do_not_use_when`。

这与“发买卖点、跟单、喊单”不同。社区层共享的是：

- 交易记忆
- 错误模式
- 复盘技能
- 适用边界

## 九、推荐交互工作流

如果你怀疑自己的理解可能有问题，可以按这个顺序使用框架：

1. 先用 `session turn` 或 `memory query` 提交自己的当前想法。
2. 看系统召回了哪些历史成功 / 失败案例。
3. 如果发现错误归因，先用 `memory revise` 修订标签、质量分和纠错说明。
4. 如果某个技能不是完全错误，而是适用范围过宽，用 `memory skill-edit` 补充禁止条件。
5. 复盘结束后重新查询同类场景，确认新的检索结果已经反映这些修正。

## 十、最终结论

所以，答案不是“框架会不会替你判卷”，而是：

- 它能让错误输入在历史证据中自然暴露。
- 它支持把错误复盘变成可以被反复召回的纠错记忆。
- 它允许用户直接局部修改标签、质量分与禁止条件。
- 它会通过 bandit 与 skillization 门槛，让错误路径越来越不活跃。

框架的立场始终是：帮助用户建立一个可追溯、可修正、可局部编辑的交易外脑，而不是假装自己拥有绝对真理。
